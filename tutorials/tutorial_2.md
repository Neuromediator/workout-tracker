# Workout Tracker — Архитектура, технологии и деплой

Обзорный документ: общая картина проекта, почему выбраны те или иные технологии, какие были альтернативы, и как всё это работает вместе в продакшене.

---

## 1. Что мы строим и зачем

**Workout Tracker** — персональное веб-приложение для планирования тренировок, логирования подходов и отслеживания прогресса. Целевая аудитория — один пользователь и несколько друзей (не массовый SaaS).

**Ключевые принципы продукта:**
- Минимум кликов во время тренировки — скорость логирования важнее всего
- Mobile-first — в зале телефон, не ноутбук
- AI-ассистент опционален — приложение полностью работает без него
- Простота важнее масштабируемости — это не стартап на миллион пользователей

Эти принципы напрямую повлияли на выбор технологий: мы выбирали **простое и дешёвое** вместо "правильного по учебнику".

## 2. Общая архитектура

```
┌─────────────────────────────────────────────────────────┐
│                      Браузер (React SPA)                │
│                                                         │
│  Supabase JS SDK        API клиент          Recharts   │
│  (аутентификация)    (fetch → /api/*)     (графики)    │
└──────────┬──────────────────┬───────────────────────────┘
           │                  │
           │ JWT токен        │ HTTPS
           ▼                  ▼
┌─────────────────┐  ┌──────────────────────────────────┐
│  Supabase Auth  │  │  Oracle Cloud VM                  │
│  (облако)       │  │                                    │
│                 │  │  Caddy (:443) → FastAPI (:8000)   │
│  - регистрация  │  │       │              │             │
│  - логин        │  │       │         SQLite (файл)     │
│  - JWT токены   │  │       │              │             │
│  - JWKS ключи   │  │  автоматический   данные на       │
│                 │  │  HTTPS            Docker volume    │
└─────────────────┘  └──────────────────────────────────┘
                                    │
                                    │ (опционально)
                                    ▼
                          ┌──────────────────┐
                          │  OpenRouter API   │
                          │  → Cerebras LLM   │
                          │  (AI-ассистент)   │
                          └──────────────────┘
```

**Три внешних сервиса:**
1. **Supabase** — аутентификация пользователей
2. **Oracle Cloud** — сервер, на котором работает приложение
3. **OpenRouter** — доступ к AI-модели (опционально)

Всё остальное — наш код, работающий в одном Docker-контейнере.

## 3. Frontend: React + Vite + TypeScript + Tailwind

### Что и зачем

| Технология | Роль | Почему именно она |
|-----------|------|-------------------|
| **React** | UI-фреймворк | Самая большая экосистема, огромное количество готовых компонентов и библиотек. Для нашего проекта подошёл бы и Vue, и Svelte — но React даёт максимум ресурсов для обучения и решения проблем |
| **Vite** | Сборщик и dev server | Мгновенный запуск в разработке (HMR за миллисекунды). Замена Webpack, который медленнее в 10-50 раз на проектах такого размера |
| **TypeScript** | Типизация | Ловит ошибки до запуска. Когда API возвращает `WorkoutSession`, TypeScript гарантирует, что мы обращаемся к существующим полям |
| **Tailwind CSS** | Стили | Utility-first: пишем `className="flex gap-2 p-4"` прямо в JSX вместо отдельных CSS файлов. Быстрее при разработке, финальный CSS содержит только используемые классы |
| **shadcn/ui** | Компоненты | Не библиотека, а набор copy-paste компонентов (Button, Dialog, Select). Код живёт в нашем проекте — полный контроль, никакой зависимости от чужих обновлений |
| **Recharts** | Графики | Декларативные React-компоненты для графиков. Простой API, хорошо работает с нашими данными прогресса |

### Альтернативы, которые мы не выбрали

| Вместо | Альтернатива | Почему не подошла |
|--------|-------------|-------------------|
| React | **Next.js** | Server-side rendering не нужен для приложения-дашборда. Next.js добавляет сложность (серверные компоненты, роутинг, middleware) без пользы для нас |
| React | **Vue.js** | Отличный фреймворк, но меньше готовых UI-библиотек. Для обучения React даёт больше transferable knowledge |
| React | **Svelte** | Меньше boilerplate, но меньше экосистема. Хороший выбор для маленьких проектов, но мы хотели React-опыт |
| Tailwind | **CSS Modules** | Больше кода, медленнее разработка. Tailwind идеален для прототипирования и маленьких команд |
| shadcn/ui | **Material UI** | Тяжёлая библиотека (~300KB), навязывает Material Design стиль. shadcn/ui — лёгкий и полностью кастомизируемый |

### Управление состоянием

Используем **`useState` + `useContext` + plain fetch** — никаких Redux, Zustand или TanStack Query.

**Почему:** для приложения с 5-7 страницами и одним пользователем глобальный state manager — overkill. Каждая страница сама загружает свои данные при монтировании. Простой `apiFetch()` wrapper — это 20 строк кода вместо конфигурации сложной библиотеки.

**Когда бы понадобился Redux/Zustand:** если бы у нас были общие данные между многими компонентами (корзина в e-commerce, чат в реальном времени) или сложные серверные состояния с кешированием.

## 4. Backend: FastAPI + Python + SQLModel

### Что и зачем

| Технология | Роль | Почему именно она |
|-----------|------|-------------------|
| **FastAPI** | Web-фреймворк | Автоматическая валидация через Pydantic, автогенерация OpenAPI документации, async support. Самый продуктивный Python-фреймворк для API |
| **Python** | Язык | Огромная экосистема для AI/ML (LiteLLM, Pydantic). Простой и читаемый. Скорость выполнения не критична для нашей нагрузки |
| **SQLModel** | ORM + схемы | Объединяет SQLAlchemy (работа с БД) и Pydantic (валидация данных) в одной модели. Один класс = таблица в БД + схема запроса/ответа API |
| **SQLite** | База данных | Файловая БД, нулевая настройка, нулевые затраты. Для одного пользователя — идеально. Файл лежит на Docker volume |
| **uv** | Менеджер пакетов | Замена pip + venv. В 10-100 раз быстрее установка зависимостей. Lock-файл гарантирует воспроизводимые сборки |

### Почему SQLModel, а не чистый SQLAlchemy или Pydantic

Обычно в FastAPI-проекте нужны **две** модели:
1. SQLAlchemy модель — описывает таблицу в БД
2. Pydantic модель — описывает формат API запроса/ответа

SQLModel объединяет их:

```python
class Exercise(SQLModel, table=True):
    id: str = Field(primary_key=True)
    name: str = Field(index=True)
    muscle_group: str
    tags: list[str] = Field(sa_column=Column(JSON))
```

Это **и** таблица в SQLite, **и** Pydantic-схема для сериализации в JSON. Меньше кода, меньше дублирования.

### Альтернативы

| Вместо | Альтернатива | Почему не подошла |
|--------|-------------|-------------------|
| FastAPI | **Django + DRF** | Django — "batteries included" фреймворк. Мощный, но тяжёлый. Для нашего API из 15 эндпоинтов — overkill. Django ORM менее гибок чем SQLAlchemy |
| FastAPI | **Express.js (Node)** | Пришлось бы писать и frontend, и backend на JavaScript. Python выбран из-за AI-экосистемы (LiteLLM) |
| FastAPI | **Go (Gin/Echo)** | Отличная производительность, но больше boilerplate. Для нашей нагрузки (5 пользователей) разница в скорости не заметна |
| SQLite | **PostgreSQL** | Надёжнее для конкурентных записей, но требует отдельный процесс/контейнер. Для одного пользователя SQLite проще и дешевле |
| SQLite | **Supabase Database** | Supabase предоставляет PostgreSQL. Можно было бы использовать и для данных, но мы хотели контроль над данными и независимость от внешнего сервиса для core-функционала |

## 5. Аутентификация: Supabase Auth

### Как это работает

```
Пользователь → Login страница → Supabase JS SDK → Supabase Cloud
                                                        │
                                                   JWT токен
                                                        │
Пользователь → API запрос (Authorization: Bearer <JWT>) → FastAPI
                                                              │
                                                    Проверка JWT подписи
                                                    (JWKS или HS256)
                                                              │
                                                    Извлечение user_id
                                                    из поля "sub"
```

**Supabase Auth** — это managed-сервис, который берёт на себя:
- Регистрацию и логин (email + пароль)
- Хранение паролей (bcrypt)
- Генерацию JWT токенов
- JWKS endpoint для верификации токенов

**Мы НЕ храним пароли.** Наш backend только проверяет JWT-подпись и извлекает `user_id`.

### Почему Supabase, а не свои пароли

Написать аутентификацию самому — одна из самых опасных задач в веб-разработке:
- Хеширование паролей (bcrypt, argon2)
- Защита от timing attacks
- Rate limiting на логин
- Email верификация
- Сброс пароля
- Хранение refresh токенов

Supabase делает всё это бесплатно (до 50,000 MAU). Мы пишем 0 строк кода для аутентификации и получаем production-grade безопасность.

### Альтернативы

| Сервис | Бесплатный тир | Почему не выбрали |
|--------|---------------|-------------------|
| **Auth0** | 7,500 MAU | Тоже отличный вариант. Supabase проще в настройке и даёт больше бесплатных пользователей |
| **Firebase Auth** | Безлимитный | Привязка к Google экосистеме. Supabase — open-source альтернатива |
| **Clerk** | 10,000 MAU | Красивые ready-made компоненты, но React-specific и закрытый |
| **Свой auth** | — | Слишком много работы и риск ошибок безопасности для проекта на 5 пользователей |

## 6. AI-интеграция: LiteLLM + OpenRouter + Cerebras

### Архитектура AI-пайплайна

```
Пользователь: "Create a push day workout"
        │
        ▼
  Frontend → POST /api/ai/chat
        │
        ▼
  Backend: ai_service.py
    1. Собирает контекст (упражнения, сессии, рутины)
    2. Формирует system prompt с JSON-схемой
    3. Отправляет в LiteLLM
        │
        ▼
  LiteLLM → OpenRouter API → Cerebras (gpt-oss-120b)
        │
        ▼
  LLM возвращает JSON:
  {
    "action": "create_session",
    "summary": "Created push day: bench press, OHP, tricep dips",
    "session_data": { ... }
  }
        │
        ▼
  Backend: парсит JSON → выполняет действие → создаёт сессию в БД
        │
        ▼
  Frontend: показывает ответ + переходит к сессии
```

### Три уровня абстракции

| Уровень | Технология | Роль |
|---------|-----------|------|
| **LiteLLM** | Python-библиотека | Единый интерфейс для 100+ LLM-провайдеров. Можно сменить модель одной строкой |
| **OpenRouter** | API-прокси | Роутит запросы к разным провайдерам. Один API-ключ — доступ к OpenAI, Anthropic, Meta и другим |
| **Cerebras** | Inference-провайдер | Специализированные чипы для LLM. Очень быстрый inference (100+ токенов/сек) |

**Зачем три уровня?** Гибкость. Если Cerebras недоступен — OpenRouter переключит на другой провайдер. Если OpenRouter подорожает — LiteLLM позволит переключиться на прямой API другого провайдера без изменения кода.

### Structured Outputs

AI не возвращает произвольный текст — он возвращает **типизированный JSON**, описанный Pydantic-моделью:

```python
class WorkoutAction(BaseModel):
    action: Literal["create_session", "delete_session", "info"]
    requires_confirmation: bool
    summary: str
    session_data: Optional[SessionData]
    session_id: Optional[str]
```

**Зачем:** вместо парсинга текста "I've created a workout for you..." мы получаем структурированные данные, которые можно напрямую использовать для создания записей в БД.

### Альтернативы

| Вместо | Альтернатива | Почему не подошла |
|--------|-------------|-------------------|
| LiteLLM | **Прямой API OpenAI/Anthropic** | Привязка к одному провайдеру. LiteLLM даёт свободу выбора |
| OpenRouter | **Прямой API Cerebras** | Менее гибко. OpenRouter даёт fallback на другие модели |
| Cerebras | **OpenAI GPT-4** | Дороже и медленнее для нашего use case. Cerebras — бесплатный тир через OpenRouter |
| Structured JSON | **Function calling** | Function calling мощнее, но сложнее в реализации. JSON-ответ + Pydantic-валидация — проще для нашего набора из 3 действий |

## 7. Стратегия деплоя

### Принцип: максимальная простота

Наша стратегия — **один сервер, один контейнер, минимум движущихся частей**.

```
GitHub (код) → SSH на сервер → git pull → docker compose up --build
```

Нет CI/CD пайплайна, нет Kubernetes, нет blue-green деплоя. Для 5 пользователей это идеально — downtime в 30 секунд во время деплоя никого не волнует.

### Роль каждого сервиса в деплое

```
┌──────────────────────────────────────────────────────────┐
│                     Продакшен                            │
│                                                          │
│  ┌─────────────┐    ┌──────────────────────────────┐    │
│  │   Caddy     │    │     App контейнер             │    │
│  │             │    │                                │    │
│  │  - HTTPS    │───▶│  FastAPI (Uvicorn)            │    │
│  │  - TLS      │    │    ├── API endpoints          │    │
│  │  - Let's    │    │    ├── Frontend (статика)      │    │
│  │    Encrypt  │    │    └── SQLite (data/)          │    │
│  └─────────────┘    └──────────────────────────────┘    │
│                                                          │
│  Oracle Cloud VM (ARM, 2 CPU, 12 GB RAM)                │
│  Ubuntu 24.04 + Docker                                   │
└──────────────────────────────────────────────────────────┘
         ▲                        ▲                ▲
         │                        │                │
    Домен (DNS)            Supabase Auth     OpenRouter AI
    Namecheap              (JWT проверка)    (опционально)
```

| Сервис | Роль | Бесплатный тир | Альтернатива |
|--------|------|---------------|--------------|
| **Oracle Cloud** | Сервер (VM) | 4 CPU, 24 GB RAM ARM — навсегда бесплатно | AWS Free Tier (750 часов/год, потом платно), Hetzner (4 EUR/мес), Railway (5$/мес), Fly.io (3 VM бесплатно) |
| **Namecheap** | Домен | — (1-3$/год) | Porkbun, Cloudflare Registrar, Google Domains |
| **Caddy** | HTTPS + реверс-прокси | Open-source | Nginx + Certbot (бесплатно, но 30+ строк конфига вместо 3), Traefik (мощнее, но сложнее) |
| **Docker** | Контейнеризация | Open-source | Podman (drop-in замена), или деплой без контейнеров (systemd + venv), но Docker удобнее |
| **Supabase** | Аутентификация | 50,000 MAU | Auth0, Firebase Auth, или свой JWT (сложнее) |
| **OpenRouter** | AI-прокси | Бесплатные модели доступны | Прямой API OpenAI/Anthropic/Cerebras |
| **GitHub** | Хранение кода | Бесплатно | GitLab, Bitbucket |

### Почему Oracle Cloud, а не другие

| Провайдер | Бесплатный тир | Подвох |
|-----------|---------------|--------|
| **Oracle Cloud** | 4 CPU, 24 GB RAM, навсегда | Сложный UI, ARM (не x86), иногда нет свободных VM |
| **AWS Free Tier** | t2.micro (1 CPU, 1 GB) | Только 12 месяцев, потом ~10$/мес |
| **Google Cloud** | e2-micro (0.25 CPU, 1 GB) | Очень слабый, трафик платный |
| **Azure** | B1s (1 CPU, 1 GB) | 12 месяцев, потом платно |
| **Hetzner** | — | 4 EUR/мес за 2 CPU, 4 GB — хороший платный вариант |

Oracle — единственный, кто даёт **мощный** сервер **навсегда бесплатно**. Для нашего масштаба — идеальный выбор.

### Почему один контейнер, а не микросервисы

Классическая архитектура разделила бы приложение на:
- Frontend контейнер (Nginx + статика)
- Backend контейнер (FastAPI)
- Database контейнер (PostgreSQL)
- Reverse proxy (Nginx/Traefik)

Мы используем **один контейнер** (FastAPI раздаёт и API, и фронтенд):

**Плюсы:**
- Проще деплой: `docker compose up` — и всё работает
- Меньше ресурсов: один процесс вместо четырёх
- Нет проблем с межконтейнерной сетью
- SQLite не требует отдельного процесса

**Минусы:**
- Нельзя масштабировать фронтенд и бэкенд независимо
- При ошибке в бэкенде падает и раздача фронтенда

Для нашего масштаба минусы нерелевантны. Масштабирование понадобится при сотнях concurrent пользователей — мы далеко от этого.

## 8. Безопасность: общий подход

### Принцип: defense in depth (эшелонированная защита)

Каждый уровень системы имеет свою защиту, и компрометация одного уровня не ломает всё:

| Уровень | Угроза | Защита |
|---------|--------|--------|
| **Сеть** | Перехват трафика | HTTPS (Caddy + Let's Encrypt), только порты 80/443 открыты |
| **Аутентификация** | Несанкционированный доступ | Supabase JWT на каждый API-запрос, подпись проверяется через JWKS |
| **Авторизация** | Доступ к чужим данным | Каждый SQL-запрос фильтруется по `user_id` из JWT |
| **Секреты** | Утечка API-ключей | `.env` не в git, переменные окружения передаются через Docker |
| **CORS** | Запросы с чужих доменов | `CORS_ORIGINS` ограничен нашим доменом |
| **Данные** | Потеря при перезапуске | SQLite файл на Docker volume |
| **Сервер** | Взлом через SSH | Доступ только по SSH-ключу (не по паролю) |

### Что мы НЕ делаем (и почему)

- **Rate limiting** — при 5 пользователях DDoS-защита не нужна. При росте — добавить через Caddy plugin или Cloudflare
- **WAF (Web Application Firewall)** — overkill для персонального приложения
- **Бэкапы БД** — SQLite файл можно скопировать через `scp`. Автоматические бэкапы добавить позже при необходимости
- **Логирование и мониторинг** — `docker compose logs` для отладки. Для серьёзного мониторинга — Grafana + Loki, но это отдельный проект

## 9. Что бы мы изменили при масштабировании

Текущая архитектура оптимизирована для простоты. Вот что пришлось бы менять при росте:

| Текущее решение | Когда менять | На что менять |
|----------------|-------------|---------------|
| **SQLite** | > 50 concurrent users | PostgreSQL (отдельный контейнер или Supabase DB) |
| **Один сервер** | > 1000 users или SLA требования | Несколько VM за Load Balancer |
| **git pull деплой** | Команда > 2 разработчиков | GitHub Actions CI/CD |
| **Docker Compose** | > 5 сервисов | Kubernetes (или Docker Swarm) |
| **Один контейнер** | Разные паттерны нагрузки на frontend/backend | Отдельные контейнеры + CDN для статики |
| **localStorage для чата** | Нужна история на разных устройствах | Хранение чат-истории в БД |
| **Нет кеширования** | Медленные ответы API | Redis для кеширования частых запросов |

**Ключевой принцип:** не оптимизировать заранее. Текущая архитектура справляется с нашей нагрузкой. Усложнять — только когда появится реальная проблема.
